Blueprint for Autonomous Engineering: Spec-Driven Development and Harness Architecture

1. The Paradigm Shift: From Manual Implementation to Agent Steering

The strategic transition from "manual coding" to a model where humans steer and agents execute represents the final industrialization of software engineering. We are moving away from the era of the "code writer"—limited by human syntax mastery and cognitive load—to the era of the "environment designer" and "intent specifier." This shift is not merely a tooling upgrade; it is a fundamental reconfiguration of the software development lifecycle (SDLC) required to achieve 10th-order increases in engineering velocity. By focusing on the scaffolding that surrounds the model rather than the lines of code within it, we transform development from a linear human struggle into a high-throughput autonomous engine.

Deconstructing the Core Constraint

The "zero manual code" philosophy is a deliberate architectural constraint. By forbidding direct human intervention in the codebase, architects are forced to prioritize the creation of robust feedback loops and legible abstractions. This transforms the role of the developer into a system architect who manages capabilities rather than syntax. If an agent fails, the architect does not "try harder" manually; they identify the missing capability in the environment and make it both legible and enforceable for the agent.

Synthesis of Efficiency Gains

Data from high-velocity autonomous environments (e.g., OpenAI’s internal 0-line manual code experiment) reveals a staggering gap between traditional and steering models:

- Traditional Development: Manual implementation of logic and tests; velocity capped by human typing; typical throughput of 0.5–1 PR per day.
- Autonomous Steering: 1/10th the time reduction to ship products; throughput of 3.5+ PRs per day per engineer; millions of lines of code managed by teams as small as three engineers.

The "Steering" Mechanism: The Ralph Wiggum Loop

Architects must enforce the Ralph Wiggum Loop to ensure human attention is never the bottleneck for syntax validation. In this cycle, humans interact via high-level prompts to define tasks. The agents then autonomously gather context, open PRs, and request specific agent reviews. The loop iterates—reviewing, revising, and validating—until all automated and inferential reviewers are satisfied. This shift in mindset necessitates a more rigid structure for documentation, which we codify as Spec-Driven Development.

2. Spec-Driven Development (SDD): Establishing the North Star

In an agent-first ecosystem, maintaining software means evolving specifications rather than tweaking lines of code. SDD is a documentation-first approach where the "lingua franca" of development moves to a higher level, treating code as a "last-mile" generated artifact.

SDD Maturity Levels

The level of autonomy is directly proportional to how the specification is anchored in the workflow:

Model Definition Autonomy vs. Control
Spec-first A comprehensive spec is written before any code generation begins. High human control; ensures initial alignment.
Spec-anchored The spec is a living document, evolving with the feature throughout its lifecycle. Balanced; enables maintenance via spec evolution.
Spec-as-source The spec is the primary artifact; humans only edit specs, never the code. Highest autonomy; code is treated as a generated byproduct.

The Spec as a Functional Artifact

High-quality specs must follow an "OpenSpec" philosophy: they are structured, behavior-oriented artifacts containing Product Context, Design Principles, and Acceptance Criteria. By aligning on these artifacts before implementation, we eliminate "context anxiety"—the tendency for agents to wrap up work prematurely as their context window fills—and ensure predictable outcomes.

Contrast of SDD Frameworks

- OpenSpec: A lightweight, artifact-guided framework. It avoids rigid phase-gates, allowing fluid updates to proposals and tasks across 25+ AI tools.
- spec-kit (GitHub): A heavyweight system using rigid phase-gates and extensive Markdown templates, best suited for formal enterprise transitions.
- Kiro (AWS): Powerful but limited by environment lock-in (VS Code) and specific model dependencies.

The Sprint Contract and Model-Awareness

Architects must bridge user stories and testable implementation via a Sprint Contract—a negotiated agreement where a Generator proposes a build plan and an Evaluator verifies it against the spec. However, this architecture must be model-aware:

- Opus 4.5/Sonnet 4.5: Requires heavy scaffolding, Sprint Contracts, and frequent Context Resets to manage context anxiety.
- Opus 4.6+: These models plan more carefully and sustain longer tasks. At this level, architects can lean out the scaffold, removing the overhead of Sprint Contracts to maximize velocity.

3. Harness Engineering: Building the Agent’s Operating Environment

Harness Engineering is the joint optimization of the model and its surrounding infrastructure (Harness = Scaffolding + Environment). A model like Claude, regardless of its reasoning capacity, fails in long-horizon tasks without a structured system that regulates its state space.

Evolutionary Layers of a Harness

1. Layer 1: Action Interface: Translating model outputs into executable calls (files, browsers, sandboxes) to establish an action-observation loop.
2. Layer 2: Workflow Infrastructure: Maintaining persistent workspace states (terminals, compilers) to allow iterative inspection and revision.
3. Layer 3: User-Centric Persistence: Managing cross-session continuity, user routines, and multi-agent coordination.

Regulation Categories and Ashby’s Law

We utilize Ashby’s Law of Requisite Variety to justify limiting agent autonomy via rigid topologies. A regulator can only control what it has a model of; by narrowing the state space through "Harness Templates," we make comprehensive regulation achievable. We categorize these controls as:

- Maintainability Harness: Regulates code quality via linters and complexity checks.
- Architecture Fitness Harness: Enforces structural characteristics (e.g., logging standards, performance requirements).
- Behavior Harness: Regulates functional correctness through E2E validation.

Feedforward (Guides) vs. Feedback (Sensors)

- Guides (Feedforward): Instructions (e.g., AGENTS.md, Skills) that steer behavior before action.
- Sensors (Feedback): Deterministic tools (tests, linters) that provide signals after action.

Computational vs. Inferential Execution

- Computational: Fast, CPU-run, deterministic (e.g., type checkers). These provide reliable millisecond-level feedback.
- Inferential: Slower, GPU-run, "LLM-as-judge" semantic reviews. Necessary for judging "taste" but too expensive to run on every commit.

4. Multi-Agent Loops: The Generator-Evaluator (GAN) Model

To prevent the "self-praising agent" failure mode, we separate "doing" from "judging" using a GAN-inspired structure. This addresses the lack of self-critical judgment inherent in single-agent sessions.

Three-Agent Persona Architecture

- The Planner: Expands 1-4 sentence prompts into ambitious product specs. It focuses on high-level design to prevent cascading errors from over-specifying granular technical details.
- The Generator: Implements features in chunks. In Opus 4.5, it requires context resets; in Opus 4.6, it can run as one continuous session with automatic compaction.
- The Evaluator/QA: A skeptical critic using tools like Playwright to exercise the UI/API. It grades against four concrete terms: Design Quality, Originality, Craft, and Functionality.

The Strategic Pivot and Creative Leap

The Generator does not just refine; it must be empowered to make Strategic Pivots. In an autonomous build of a Dutch Art Museum site, an evaluator's critique of a "standard" layout led the generator to scrap its approach and reimagine the site as a 3D spatial experience. This "creative leap" is a key architectural capability unlocked by the GAN loop.

5. System of Record: Making the Repository "Agent-Legible"

The core principle: If it isn’t in the repo, it effectively doesn’t exist to the agent. Knowledge in Slack or human heads is "unseen knowledge" that leads to drift.

Anatomy of an Agent-Legible Knowledge Base

We organize the docs/ directory to serve as the singular system of record for all agents, including specialized ones like Aardvark.

docs/
├── design-docs/ # Core beliefs and architectural principles
├── exec-plans/ # Active and completed implementation logs
│ ├── active/
│ └── completed/
├── product-specs/ # Structured requirements and acceptance criteria
└── references/ # Design system and runtime reference files

AGENTS.md as a "Map"

AGENTS.md is not an encyclopedia; it is a ~100-line table of contents. It points the agent to deeper sources of truth (e.g., ARCHITECTURE.md), preventing a monolithic file from rotting or crowding out the task context.

Enforcing Taste Invariants

We enforce architectural integrity via custom, agent-generated linters. A non-negotiable invariant is the forward-only dependency rule: Types → Config → Repo → Service → Runtime → UI Cross-cutting concerns (auth, telemetry) enter only through explicit Providers.

Doc-Gardening: Automating the Friday Cleanup

Historically, teams spent 20% of their time (Fridays) cleaning "AI slop." We have replaced this with Doc-Gardening agents. These background tasks scan for stale docs, update quality grades, and prune debt, ensuring the knowledge base reflects real code behavior.

6. Implementation Blueprint: Leveraging Claude and MCP

Architects must map these frameworks to Claude’s ecosystem using the Model Context Protocol (MCP) to expose "Sensors" directly to the reasoning engine.

- The Victoria Observability Stack: We give agents a full observability suite using Victoria Logs, Victoria Metrics, and Victoria Traces. By exposing LogQL and PromQL APIs, prompts like "ensure service startup < 800ms" become executable.
- Bootable Worktrees: We enable the "bootable Worktree" strategy, allowing Claude to launch isolated app instances per change to validate fixes via Chrome DevTools MCP snapshots and DOM navigation.
- Claude Code Slash Commands: The SDD workflow is operationalized via commands: /opsx:propose (Planning), /opsx:verify (QA Loop), and /opsx:onboard (Refreshing agent instructions).

7. Strategic Outlook: Scaling and Maintaining Autonomous Velocity

The engineer's value now lies in the discipline of the scaffolding. We prioritize a "minimal blocking gate" philosophy: corrections are cheaper than waiting. PRs are short-lived, and flakes are addressed with follow-up agent runs rather than blocking the pipeline.

The New Engineer’s Checklist

1. Prioritize Work: Translate user needs into acceptance criteria.
2. Translate Feedback: Convert subjective critiques into objective, gradable terms.
3. Identify Missing Capabilities: When agents struggle, build the tool or guardrail that enables success next time.

The Three Pillars of Autonomous Engineering

- Rigid Architecture: Enforcing strict boundaries (Types → Config → Repo → Service → Runtime → UI) to prevent drift.
- Spec-Driven Intent: A document-first approach where specs are the primary artifact.
- Multi-Agent Feedback Loops: GAN-inspired structures that separate implementation from skeptical evaluation.
